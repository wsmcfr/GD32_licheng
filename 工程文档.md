# GD32F470VET6 主控板工程详细文档

## 1. 工程概述

本工程基于 **MICU GD32F470VET6 V2** 主控板，使用 GD32F4xx 标准外设库开发，采用 AC6 (ARMCLANG) 编译器。当前工程已按官方 `BootLoader_Two_Stage` 例程规划为 **BootLoader App**，链接地址为 `0x0800D000`，用于由 BootLoader 跳转运行。

### 1.1 硬件信息

| 项目 | 参数 |
|------|------|
| 芯片型号 | GD32F470VE |
| 内核 | Cortex-M4, 最高 200MHz |
| Flash | 512KB (0x08000000 - 0x08080000) |
| SRAM | 192KB (0x20000000 - 0x20030000) + 64KB (0x10000000 - 0x10010000) |
| 调试接口 | SWD (PA13/PA14) |

### 1.1.1 BootLoader Flash 分区

| 分区 | 起始地址 | 大小 | 作用 |
|------|----------|------|------|
| BootLoader 区 | 0x08000000 | 48KB | 存放官方 BootLoader，MCU 复位后先从这里启动 |
| 参数区 | 0x0800C000 | 4KB | 存放升级标志、App 大小、CRC、入口地址等参数 |
| App 区 | 0x0800D000 | 0x63000 | 当前工程链接和运行地址，BootLoader 跳转到这里 |
| 下载缓存区 | 0x08070000 | 64KB | 当前工程用于暂存 USART2 分包接收的新 App 镜像 |

### 1.2 板载资源

- 6 路用户 LED (PD10-PD15, 高电平点亮)
- 7 路用户按键 (KEY1-KEY6 + KEYW 唤醒键)
- 0.91 英寸 SSD1306 OLED (I2C0, PB8/PB9)
- GD25Qxx SPI Flash (SPI0, PB3-PB5, PA15)
- GD30AD3344 SPI ADC (SPI3, PE12-PE14, PE10)
- SDIO SD 卡接口 (PC8-PC12, PD2)
- 4 路串口 (USART0/1/2/5)
- 2 路 ADC 通道 (PC0/PC2)
- 1 路 DAC 输出 (PA4)
- RTC 实时时钟 (外部 32.768kHz 晶振)
- 以太网 RMII 接口 (预留)

## 3. USER 目录文件详解

### 3.1 main.c - 程序入口

**文件路径**: `USER/src/main.c`

**功能**: 系统主函数，负责初始化所有外设并启动任务调度器。

**启动流程**:

```
1. systick_config()          - 配置 SysTick 定时器 (1ms 中断)
2. init_cycle_counter()      - 初始化性能计数器
3. delay_ms(200)             - 等待 200ms (防止 SWD 引脚被复用为 GPIO 时无法下载)
4. bsp_led_init()            - 初始化 6 路 LED GPIO
5. bsp_btn_init()            - 初始化 7 路按键 GPIO
6. bsp_oled_init()           - 初始化 I2C0 + DMA (OLED 接口)
7. bsp_gd25qxx_init()        - 初始化 SPI0 (SPI Flash 接口)
8. bsp_usart_init()          - 初始化 USART0 调试串口 + USART1/RS485
9. bsp_usart2_init()         - 初始化 USART2 OTA 专用串口 (460800)
10. uart_ota_reset_runtime() - 清空 OTA 运行态
11. bsp_gd30ad3344_init()    - 初始化 SPI3 (外部 ADC 接口)
12. bsp_adc_init()           - 初始化 ADC0 (内部 ADC, 2 通道 DMA)
13. bsp_dac_init()           - 初始化 DAC0 + TIMER5 (DAC 输出)
14. bsp_rtc_init()           - 初始化 RTC 实时时钟；若 VBAT 备份域有效则恢复已有时间
15. sd_fatfs_init()          - 初始化 SD 卡中断
16. app_btn_init()           - 初始化 ebtn 按键库
17. OLED_Init()              - 初始化 SSD1306 OLED 显示屏
18. lfs_storage_self_test()  - 默认执行 SPI Flash LittleFS 挂载/文件读写自检
19. test_spi_flash()         - 可选裸 Flash 读写测试，默认关闭，避免破坏 LittleFS
20. scheduler_init()         - 初始化任务调度器
21. while(1) scheduler_run() - 主循环运行任务调度
```

**printf 重定向**:
- 将 `fputc()` 重定向到 USART0，实现 `printf()` 输出到调试串口
- GCC 环境下使用 `__io_putchar()` 重定向

### 3.2 gd32f4xx_it.c - 中断服务函数

**文件路径**: `USER/src/gd32f4xx_it.c`

**功能**: 定义所有中断服务函数 (ISR)。

| 中断函数 | 触发源 | 功能 |
|----------|--------|------|
| `NMI_Handler` | NMI 异常 | 死循环 (调试用) |
| `HardFault_Handler` | 硬件错误 | 死循环 (调试用) |
| `MemManage_Handler` | 内存管理错误 | 死循环 |
| `BusFault_Handler` | 总线错误 | 死循环 |
| `UsageFault_Handler` | 用法错误 | 死循环 |
| `SVC_Handler` | 系统服务调用 | 死循环 |
| `DebugMon_Handler` | 调试监视器 | 死循环 |
| `PendSV_Handler` | 可挂起系统调用 | 死循环 |
| `SysTick_Handler` | SysTick 定时器 | 调用 `delay_decrement()` 递减延时计数器 |
| `USART0_IRQHandler` | USART0 空闲中断 | DMA 接收普通调试/透传数据，设置 `rx_flag` 标志 |
| `USART1_IRQHandler` | USART1 空闲中断 | DMA 接收 RS485 数据，设置 `rs485_rx_flag` 标志 |
| `USART2_IRQHandler` | USART2 空闲中断 | DMA 接收 OTA 数据，设置 `uart_ota_rx_flag` 标志 |
| `EXTI0_IRQHandler` | EXTI0 外部中断 | 清除中断标志 (唤醒键) |
| `SDIO_IRQHandler` | SDIO 中断 | 调用 `sd_interrupts_process()` 处理 SD 卡中断 |

**USART0 接收流程**:
1. 检测到空闲帧中断 (IDLE)
2. 读取 DR 寄存器清除 IDLE 标志
3. 关闭 DMA 通道
4. 计算接收到的数据长度
5. 将本次 IDLE 帧数据复制到 `uart_dma_buffer`
6. 记录 `uart_dma_length`，并设置 `rx_flag = 1` 通知应用层
7. 重新配置 DMA 继续接收

**USART2 OTA 接收流程**:
1. 检测到 `USART2` 空闲帧中断 (IDLE)
2. 读取 DR 寄存器清除 IDLE 标志
3. 关闭 DMA 通道
4. 计算接收到的 OTA 帧长度
5. 将本次 IDLE 帧数据复制到 `uart_ota_dma_buffer`
6. 记录 `uart_ota_dma_length`，并设置 `uart_ota_rx_flag = 1` 通知 `uart_ota_task()`
7. 重新配置 DMA 继续接收下一帧 OTA 数据

## 4. APP 目录文件详解

### 4.1 scheduler.c/h - 任务调度器

**文件路径**: `APP/scheduler.c`, `APP/scheduler.h`

**功能**: 实现一个基于时间片的简易协作式任务调度器。

**调度器架构**:

```c
typedef struct {
    void (*task_func)(void);  // 任务函数指针
    uint32_t rate_ms;         // 执行周期 (ms)
    uint32_t last_run;        // 上次执行时间 (ms)
} task_t;
```

**注册的任务列表**:

| 任务函数 | 执行周期 | 功能 |
|----------|----------|------|
| `led_task` | 1ms | LED 显示刷新 |
| `adc_task` | 100ms | ADC 数据更新 |
| `oled_task` | 10ms | OLED 显示刷新 |
| `btn_task` | 5ms | 按键扫描处理 |
| `uart_task` | 5ms | USART0/RS485 普通透传处理 |
| `uart_ota_task` | 5ms | USART2 OTA 分包处理 |
| `rtc_task` | 500ms | RTC 时间显示更新 |

**调度原理**:
1. `scheduler_init()`: 计算任务数量
2. `scheduler_run()`: 遍历任务表，检查当前时间是否达到任务执行周期
3. 使用 `get_system_ms()` 获取系统毫秒时间戳
4. 到期任务立即执行，更新 `last_run` 时间戳

**添加新任务方法**:
1. 在 `APP/` 目录创建 `xxx_app.c/h`
2. 实现任务函数 `void xxx_task(void)`
3. 在 `scheduler.c` 的 `scheduler_task[]` 数组中添加条目

### 4.2 led_app.c/h - LED 显示任务

**文件路径**: `APP/led_app.c`, `APP/led_app.h`

**功能**: 控制 6 路 LED 的显示状态。

**全局变量**:
- `ucLed[6]`: LED 状态数组，1 = 亮，0 = 灭。初始值 `{1,0,1,0,1,0}` (交替亮灭)

**函数说明**:

| 函数 | 功能 |
|------|------|
| `led_disp(uint8_t *ucLed)` | 根据状态数组控制 LED，带防重复写入优化 |
| `led_task()` | 调度器调用的任务函数 |

**LED 控制逻辑**:
- 遍历 `ucLed[6]` 数组，将状态编码为 6 位 `temp` 变量
- 与上次状态 `temp_old` 比较，仅在状态变化时更新 GPIO
- 通过 `LED1_SET()` ~ `LED6_SET()` 宏控制对应引脚

**函数说明**:

| 函数 | 功能 |
|------|------|
| `oled_printf(x, y, format, ...)` | 类似 printf 的 OLED 显示函数，支持格式化字符串 |
| `oled_task()` | 调度器调用的任务函数，更新 OLED 显示内容 |

**实现细节**:
- 使用 `vsnprintf()` 格式化字符串到 512 字节缓冲区
- 调用 `OLED_ShowStr()` 显示 6x8 ASCII 字符
- X 轴范围: 0-127, Y 轴范围: 0-3 (共 4 行)

### 4.5 usart_app.c/h - 串口收发任务

**文件路径**: `APP/usart_app.c`, `APP/usart_app.h`

**功能**: 提供串口打印，以及 `USART0 <-> RS485` 的普通透明转发功能。

**全局变量**:
- `rx_flag`: 接收完成标志 (在 ISR 中置 1)
- `uart_dma_buffer[1024]`: USART0 单帧接收缓冲区，用于普通调试/透传数据
- `uart_dma_length`: 本次 USART0 IDLE 帧的有效字节数

**函数说明**:

| 函数 | 功能 |
|------|------|
| `my_printf(usart_periph, format, ...)` | 支持任意串口的 printf 函数 |
| `uart_task()` | 检查 `rx_flag` / `rs485_rx_flag`，完成 USART0 与 RS485 双向透明转发 |

**my_printf 实现**:
1. 使用 `vsnprintf()` 格式化字符串到 512 字节缓冲区
2. 逐字节发送到指定串口
3. 等待发送完成 (TBE 标志)
4. 返回实际发送长度

### 4.6 uart_ota_app.c/h - OTA 升级任务

**文件路径**: `APP/uart_ota_app.c`, `APP/uart_ota_app.h`

**功能**: 提供 `USART2` 专用 OTA 分包接收、ACK 回包、下载缓存区写入和 BootLoader 交接流程。

**全局变量**:
- `uart_ota_rx_flag`: OTA 接收完成标志 (在 `USART2_IRQHandler` 中置 1)
- `uart_ota_dma_buffer[1024]`: USART2 单帧 OTA 缓冲区，可容纳 `512B` DATA 载荷加协议头
- `uart_ota_dma_length`: 本次 USART2 IDLE 帧的有效字节数

**函数说明**:

| 函数 | 功能 |
|------|------|
| `uart_ota_reset_runtime()` | 清空 OTA 会话状态、共享长度和私有缓冲区 |
| `uart_ota_task()` | 处理 START/DATA/END 帧、回 ACK，并在成功后软件复位交给 BootLoader |

**OTA 处理逻辑**:
1. `USART2_IRQHandler` 只负责把一帧完整 OTA 数据移交给任务层
2. `uart_ota_task()` 先校验 START 帧头，再擦除下载缓存区
3. 每个 DATA 分包按 `offset` 直接写入 `0x08070000`
4. 首个 DATA 分包额外校验 App 向量表合法性
5. END 阶段同时校验运行时 CRC 和下载区回读 CRC
6. 校验通过后调用 `bootloader_port_write_upgrade_info()` 写参数区并软件复位

**深度睡眠流程**:
1. 关闭所有串口 DMA 和中断
2. 关闭 OLED 电荷泵和显示输出，再关闭 I2C
3. 让 GD25QXX 进入 deep power-down、让 GD30AD3344 切到掉电/单次转换模式，再关闭 SPI 总线
4. 关闭 SDIO 和 SD 卡
5. 关闭 ADC、DAC、定时器
6. 将所有 GPIO 配置为模拟输入 (最低漏电)
7. 配置唤醒键 EXTI 中断
8. 关闭不再需要的 USART / SPI / I2C / DMA / ADC / DAC / TIMER / GPIO 时钟门控
9. 清除 PMU 标志
10. 完整停止 SysTick 计数器和中断
11. 调用 `pmu_to_deepsleepmode(PMU_LDO_LOWPOWER, PMU_LOWDRIVER_ENABLE, WFI_CMD)` 进入深度睡眠
12. 唤醒后重新初始化所有外设，并在 SPI 总线恢复后释放 GD25QXX 的 deep power-down
13. 初始化完成后关闭 `EXTI0` 唤醒中断，只保留普通按键输入功能

**RTC / VBAT 行为说明**:
- 当前板上 `RTC` 使用外部 `32.768kHz` 晶振。
- 若 `VBAT` 端存在纽扣电池供电，且备份域标记 `RTC_BKP0 == BKP_VALUE` 仍有效，则主电掉电后再次上电会继续沿用原来的 RTC 时间。
- 若 `VBAT` 未供电，或备份域从未初始化/已丢失，则 `bsp_rtc_init()` 会按工程默认时间重新建表。

- 低功耗策略: 深度睡眠前发送 `deep power-down` 指令，唤醒后在 SPI0 恢复完成后发送 `release` 指令

**LittleFS 分区与安全策略**:

| 项目 | 当前值 | 说明 |
|------|------|------|
| 物理 Flash 容量 | `2MB` | `LFS_FLASH_TOTAL_SIZE`，按 GD25Q16 配置 |
| LittleFS 管理区 | `0x000000` ~ `0x1FEFFF` | `LFS_FLASH_FS_SIZE`，共 511 个 4KB 块 |
| 裸测试保留区 | `0x1FF000` ~ `0x1FFFFF` | `LFS_FLASH_RAW_TEST_START_ADDR`，不纳入 LittleFS |
| LittleFS 自检 | 默认启用 | `LFS_STORAGE_BOOT_SELF_TEST_ENABLE=1` |
| 裸 Flash 测试 | 默认关闭 | `SPI_FLASH_RAW_TEST_ENABLE=0` |

启动时默认运行 `lfs_storage_self_test()`。该测试会先挂载 LittleFS；如果首次使用或元数据无效，则只格式化 LittleFS 管理区。挂载成功后优先只读校验 `/boot_check.txt`，文件缺失或内容不匹配时才写入修复，避免每次启动都增加 Flash 擦写次数。`test_spi_flash()` 保留给底层 GD25QXX 驱动排障，启用前必须确认它只擦写末尾保留扇区，不能改回 `0x000000`。

**预期串口输出**:

首次空片、文件系统损坏或测试文件缺失时，预期会看到格式化或写入修复日志：

```text

LFS: mount failed (...), formatting fs area...
LFS: format and mount ok
LFS: self-test file missing or unreadable (-2), rewrite
LFS: self-test PASS: /boot_check.txt=GD25QXX_LITTLEFS_BOOT_CHECK
BOOT: test_spi_flash skipped (SPI_FLASH_RAW_TEST_ENABLE=0)
```

后续启动文件内容正确时，预期只读通过，不再写入测试文件：

```text
LFS: self-test start
LFS: mount ok, size=2044KB, blocks=511
LFS: self-test PASS(read-only): /boot_check.txt=GD25QXX_LITTLEFS_BOOT_CHECK
BOOT: test_spi_flash skipped (SPI_FLASH_RAW_TEST_ENABLE=0)
```

## 8. 使用说明

### 8.1 编译

1. 打开 `MDK/2026706296.uvprojx`
2. 执行 `Project -> Rebuild all target files`
3. 确保安装了 `GigaDevice.GD32F4xx_DFP.3.0.3` Pack
4. 当前工程已打开 HEX 输出，并在构建后调用 `fromelf` 生成 `MDK/output/Project.bin`

| 输出文件 | 用途 |
|------|------|
| `MDK/output/Project.hex` | 带地址信息，可用于烧录工具直接写入 App 区 `0x0800D000` |
| `MDK/output/Project.bin` | 纯二进制 App 镜像，用于 USART2 分包 OTA 发送，不要直接用串口助手发送 |
| `MDK/output/Project.uota` | 旧完整包格式，仍可生成用于离线检查；当前低 RAM OTA 推荐使用 `--mode send` 分包发送 |

### 8.1.1 AC6 脱机运行检查

当前工程使用 ARMCLANG / AC6 编译，App 必须禁用 C 库 semihosting，并由 `USER/main.c` 提供 `_sys_open()`、`_sys_write()`、`_sys_exit()`、`_ttywrch()` 和 `fputc()` 等 retarget 桩函数。

| 检查项 | 正确状态 | 作用 |
|---|---|---|
| `__use_no_semihosting` | `MDK/Listings/Project.map` 中应出现该符号 | 告诉 Arm C 库不要通过调试器执行主机文件/控制台服务 |
| `_sys_open` / `_sys_write` / `_sys_exit` / `_ttywrch` | 符号来源应为 `main.o` | 避免链接到 C 库默认 semihosting 桩函数 |
| 调试器反汇编 | 不应停在 `BKPT 0xAB` | `BKPT 0xAB` 是 semihosting 请求，脱机运行时通常会 HardFault |
| 脱机串口日志 | BootLoader `jump app` 后应继续出现 `BOOT: handoff start` | 证明 App 已经进入自己的初始化流程 |

如果串口只打印到 `BootLoader : jump app vtor:0x0800d000 ...`，但 Keil 调试模式点继续几次后能运行，优先检查 App 是否重新引入了 semihosting，而不要先修改 BootLoader 跳转地址或厂家 `system_gd32f4xx.c`。

### 8.2 下载

1. 首次调试时，先将官方 BootLoader 烧录到 `0x08000000`
2. 当前工程编译出的 App 镜像链接地址为 `0x0800D000`
3. 通过 BootLoader 下载/搬运 App 时，App 必须写入 `0x0800D000`
4. 若临时使用 SWD 直接下载当前工程，需要确认不会擦除 `0x08000000` 的 BootLoader 区

注意：当前工程内复制的 `BootLoader_Two_Stage\27_0_BootLoader` 已针对本 App 做过优化，不再使用官方原始的 `CONFIG_APP_SIZE 1024*20` 整包缓存，也不再固定只擦除 App 开头 `3 * 4KB`。现在 BootLoader 会按 `appSize` 计算擦除页数，并以 1024B 分块从 `0x08070000` 搬运到 `0x0800D000`。当前 App 已实现 `USART2` 分包在线升级接收：App 每收到一个 DATA 帧就写下载缓存区，END 校验通过后写参数区并复位，BootLoader 再执行搬运和 CRC 校验；`USART0` 仅保留日志和普通透传。

### 8.2.1 USART2 在线升级

1. 每次重新编译 App 后，在工程根目录先检查分包信息：

```powershell
cd D:\GD32\2026706296
python tools\make_uart_ota_packet.py --mode stream-info --version 0x00000006 --chunk-size 512
```

2. 使用 ACK 分包模式发送，`COM29` 按实际接到 `USART2` 的串口号替换；虽然工具默认已经是 `460800`，这里仍建议显式写出波特率，避免和旧流程混淆：

```powershell
python tools\make_uart_ota_packet.py --mode send --port COM29 --baudrate 460800 --version 0x00000006 --chunk-size 512
```

3. 发送前先确认 `--version` 比上一次升级版本更大，避免参数区里旧版本和新版本混淆。
4. 不要直接发送 `Project.bin`、`Project.hex` 或旧完整包；当前低 RAM OTA 需要 START/DATA/END 帧和 ACK 节流。
5. 发送工具会先打印固件摘要，然后依次打印 `START acked`、多行 `DATA acked`、最后 `END acked`，用于观察进度和定位卡帧位置。
6. App 收到 START 后擦除下载缓存区，每收到 DATA 后立即写入 `0x08070000`，END 校验通过后会打印 `OTA: ready, reset to BootLoader`。
7. BootLoader 随后应打印 `app crc32 check pass`、`app update success`，并在再次启动时显示 `updateStatus:0x00`、`updateFlag:0x00`。

典型发送输出示例：

```text
send stream MDK\output\Project.bin: firmware=33536 bytes, crc=0xC0B85342, version=0x00000006, chunk_size=512, chunks=66, channel=USART2, port=COM29, baudrate=460800
START acked: chunk=0/66, frames=1/68, bytes=0/33536 (0%)
DATA acked: chunk=1/66, frames=2/68, bytes=512/33536 (1%)
...
DATA acked: chunk=66/66, frames=67/68, bytes=33536/33536 (100%)
END acked: chunk=66/66, frames=68/68, bytes=33536/33536 (100%)
sent stream frames=68, channel=USART2, port=COM29, baudrate=460800
```

流式 OTA 帧格式如下：

| 帧 | 固定字段 | 说明 |
|------|------|------|
| START | `0xA55A5AA5, type=1, appVersion, firmwareSize, firmwareCRC32, headerCRC32` | App 校验后擦除 `0x08070000` 下载缓存区 |
| DATA | `0xA55A5AA5, type=2, seq, offset, length, chunkCRC32, chunk` | App 校验序号、偏移和 CRC 后立即写入 Flash |
| END | `0xA55A5AA5, type=3, firmwareSize, firmwareCRC32` | App 读回校验下载缓存区，写参数区并复位 |
| ACK | `0xA55A5AA5, type=0x80|原帧类型, status, value0, value1` | 上位机收到成功 ACK 后才发送下一帧 |

限制：当前下载缓存区为 `64KB`，因此 `Project.bin` 不能超过 `64KB`。本次规划把原来 App 区和下载缓存区之间的 12KB 间隙并入下载缓存区，解决加入 LittleFS 后 `Project.bin` 超过 52KB 的问题；若后续 App 超过 64KB，需要继续重新规划 Flash 分区或改为外部 Flash/SD 卡缓存升级。

### 8.3 串口调试

1. 连接 USART0 (PA9-TX, PA10-RX)
2. 波特率: 460800, 8N1
3. 上电后会打印启动日志
4. `USART2` (PD8-TX, PD9-RX) 为 OTA 专用口，升级时请把上位机发送端接到这一组引脚
5. `BOOT: start` 只会出现在 `USART0`；`USART2` 上电后只会额外输出一次 `OTA2: ready` 作为 OTA 接线探测串

如果 `--mode send` 卡在 `timeout waiting ack for frame type 1`，请按下面顺序观察：

| 顺序 | 观察口 | 期望现象 |
|---|---|---|
| 1 | `USART2` (`PD8/PD9`) | 板子上电时能看到一次 `OTA2: ready` |
| 2 | `USART0` (`PA9/PA10`) | 开始发送后出现 `OTA: rx irq=... len=... magic=... type=...` |
| 3 | `USART0` (`PA9/PA10`) | 若只有 `OTA: rx ...` 没有上位机 ACK，说明 RX 已通，优先查 `PD8` 发射链路或串口收线 |
| 4 | 两路都无输出 | 优先查是否仍在运行旧 App，或接线/串口号并未落在 `USART2` |

## 9. 注意事项

1. **DMA 通道冲突**: 使用新外设前请检查 DMA 通道映射，避免冲突
2. **深度睡眠**: 按 KEY2 会进入深度睡眠，需按 KEYW (PA0) 唤醒
3. **OLED 刷新率**: OLED 任务 10ms 执行一次，避免过快刷新导致 I2C 总线拥堵
4. **SD 卡**: SD_FATFS_DEMO_ENABLE 宏控制是否启用 SD 卡测试
5. **printf 缓冲区**: `my_printf` 和 `oled_printf` 内部使用 512 字节缓冲区，格式化字符串不能超过此长度
